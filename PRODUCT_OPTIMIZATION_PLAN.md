# MathAnim Desktop 产品优化清单 + 开发排期建议

## 1. 文档目的

这份文档用于统一当前阶段的产品优化方向，回答 4 个问题：

1. 这个项目当前最影响体验的瓶颈是什么。
2. 已经完成了哪些高优先级优化。
3. 接下来应该先做什么，后做什么。
4. 如何把“能跑”继续升级成“稳定、清晰、可复盘”的桌面产品。

---

## 2. 当前产品判断

### 2.1 当前核心问题

项目当前的主要问题，不是“功能太少”，而是“复杂任务一旦失败，用户不知道系统正在做什么，也不知道下一步该怎么处理”。

具体表现为：

- 多步 AI 生成链路一长，渲染失败率明显上升。
- 失败后只能翻长日志，缺少直观诊断。
- 用户难以分辨草稿任务、排队任务、执行中任务、失败任务。
- 保底模式虽然有价值，但以前提示不够明显。
- 左侧操作区和右侧输出区的语义还不够统一。

### 2.2 产品目标

现阶段更合理的目标不是继续堆新功能，而是优先把以下 3 件事做好：

- 过程可见：用户能看到系统当前处于哪一步。
- 失败可懂：失败后能快速知道原因和修补方向。
- 状态清楚：任务的草稿、执行、失败、保底都能一眼识别。

---

## 3. 已完成优化总览

截至本轮，已经完成一组高优先级优化，重点集中在“复杂任务稳定性”和“前端可理解性”。

### 3.1 后端 / 工作流已完成

- 增加 Manim 失败诊断分类器，能识别 LaTeX、MathTex API、坐标轴标签、3D、布局复杂度等常见失败类型。
- 增强修补提示词策略，让 AI 修补更偏向稳定生成而不是继续冒险。
- 增加保底模式开关，允许在多次失败后切换到简化模板生成。
- 将失败摘要、修补建议、保底状态持久化到任务模型。
- 工作流重试时会带上失败诊断与修补提示，而不是盲修。

### 3.2 前端已完成

当前一共完成了 **10 项前端可见改动**，主要集中在创建页、输出区、任务列表三块。

1. 增加右侧任务日志面板。
2. 增加“保底模式中”状态徽标。
3. 保底任务在任务列表中高亮显示。
4. 输出状态栏改成更易懂的人话状态。
5. 运行中状态支持显示“处理中 / 保底模式中 / 保底排队中”。
6. 草稿任务单独显示为“草稿任务”。
7. 失败任务单独显示为“生成失败”。
8. 失败任务在任务列表中追加红色高亮样式。
9. 输出状态栏和保底徽标支持 hover 诊断 tooltip。
10. 输出区新增固定“失败诊断卡片”，直接展示失败原因和修补建议。

### 3.3 文档已完成

- README 已重写，项目说明更清楚。
- 本优化计划文档已重构为最新版本。

## 4. 本轮前端变化明细

这一部分专门给你复盘“前端到底改了多少、改在哪”。

### 4.1 改动数量

- 前端可见改动：10 项
- 涉及主要文件：3 个
- 相关 UI 逻辑文件：1 个

### 4.2 涉及文件

- `src/main/resources/fxml/MainView.fxml`
- `src/main/resources/css/mathanim-theme.css`
- `src/main/java/com/mathanim/ui/MainViewController.java`

说明：

- 严格说 `MainViewController.java` 属于 UI 控制层，不是纯静态前端资源，但它承担了本项目大量界面状态逻辑，因此这里一起纳入前端统计。

### 4.3 用户实际能感知到的变化

- 看到任务时，不再只是“黑箱运行”，而是能知道系统在哪个阶段。
- 看到失败时，不再只剩一大段日志，而是有固定诊断信息。
- 看到保底模式时，不再容易忽略。
- 看到任务列表时，失败任务和保底任务更容易扫到。
- 看到右侧状态栏时，更容易理解当前选中的任务到底是什么状态。

## 5. 当前优先级排序

### P0：必须继续做

这些是直接影响可用性的工作，优先级最高。

1. 左侧操作区文案统一
2. 操作按钮语义分层
3. 失败任务复跑路径更顺
4. 复杂任务失败后的自动降级策略再加强

### P1：强烈建议做

这些会明显提升产品感和可维护性。

1. 任务详情抽屉或独立详情区
2. 历史任务筛选与分组
3. 失败任务一键重试 / 一键保底重试
4. 阶段耗时与执行快照记录

### P2：中期增强

这些更偏向产品成熟度和长期复用。

1. 任务标签与收藏
2. 参考案例复用
3. 输出版本对比
4. 更多自动化测试覆盖 UI 状态逻辑

## 6. 下一阶段开发排期建议

下面这个排期不是“理想状态”，而是结合当前代码结构和实际收益排出来的。

### 第 1 周：把任务语义彻底讲清楚

目标：

- 用户不用看代码和日志，也能分辨当前任务处于什么状态。

建议内容：

- 统一左侧按钮文案。
- 区分“保存草稿”“立即生成并执行”“执行选中任务”“修改后执行”。
- 给失败任务增加“重试”和“保底重试”入口。
- 让状态栏、任务列表、日志标题三处使用统一术语。

交付结果：

- 主界面语义更加一致。
- 用户首次使用时更不容易误解操作。

### 第 2 周：把失败修补链路做成真正的产品能力

目标：

- 失败后不只是“报错”，而是能指导用户快速再试一次。

建议内容：

- 增加“一键按保底模式重试”。
- 记录每轮修补失败摘要。
- 在任务详情中展示“第几轮修补、为什么失败、这次怎么修”。
- 提供更明确的失败类别标签。

交付结果：

- 用户能够复盘。
- AI 修补体验更像一个完整闭环。

### 第 3 周：把历史任务变成资产

目标：

- 让历史任务不只是列表，而是可复用的素材库。

建议内容：

- 增加筛选：全部 / 草稿 / 成功 / 失败 / 保底。
- 增加按时间、任务类型、输出类型分组。
- 增加收藏和备注。
- 支持从历史任务快速复制输入重新生成。

交付结果：

- 高频用户效率显著提升。
- 历史记录开始具备复用价值。

### 第 4 周：补结构与测试

目标：

- 降低后续迭代成本。

建议内容：

- 拆分 `MainViewController` 的状态逻辑。
- 为状态映射、失败诊断、保底切换增加更多测试。
- 将一部分 UI 状态逻辑抽成单独 helper。

交付结果：

- 代码更稳。
- 后续改 UI 时不容易牵一发动全身。

## 8. 当前最推荐的下一步

如果只继续做一件事，我建议优先做：

### 任务操作区语义重构

也就是把左侧操作区彻底整理成下面这套逻辑：

- 保存草稿
- 预览问题拆解
- 立即生成并执行
- 执行选中任务
- 修改脚本后执行
- 失败后一键重试
- 失败后一键保底重试

原因很简单：

- 这会直接提升第一次使用的理解成本。
- 这会降低“我点了这个按钮到底会发生什么”的疑惑。
- 这会把现有能力真正组织成产品，而不是功能堆叠。

## 9. 一句话结论

MathAnim Desktop 当前最值得继续投入的方向，不是再加更多新功能，而是把“多步 AI 生成 + 失败修补 + 任务状态”这条主链路继续打磨到足够清楚、足够稳定、足够可复盘。
